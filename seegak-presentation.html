Seegak이란?

Seegak(시각)은 대규모 생물학 데이터셋의 인터랙티브 시각화를 위해 설계된 고성능 TypeScript 라이브러리입니다.

⚡

GPU 가속 렌더링

WebGL2 기반으로 100만+ 데이터 포인트를 60 FPS로 렌더링합니다. 셰이더에서 직접 컬러 팔레트 텍스처를 샘플링하여 CPU 병목을 제거합니다.

WebGL2 · GLSL ES 3.0 · Palette Texture
🧵

Worker 기반 비동기 처리

다운샘플링, 클러스터링, 통계 계산을 Web Worker에서 처리하여 메인 스레드 블로킹 없이 대규모 데이터를 다룹니다.

Reservoir Sampling · Grid Clustering · Transferable
⚛

React 네이티브 통합

React 18 컴포넌트로 래핑되어 선언적 API로 사용 가능합니다. useRef로 인스턴스에 직접 접근하여 명령형 업데이트도 지원합니다.

렌더링 파이프라인

데이터 입력부터 화면 출력까지의 처리 흐름

1

Data Input

Float32Array x, y 좌표 + 클러스터 라벨/컬러/수치값 수신

→
2

Worker Processing

다운샘플링, 정규화, 통계 계산을 Web Worker에서 비동기 수행

→
3

GPU Upload

x, y 분리 버퍼 + Uint16 인덱스 + 256x1 팔레트 텍스처 업로드

→
4

Shader Render

GLSL ES 3.0 셰이더가 팔레트 텍스처에서 색상 샘플링 후 렌더링

→
5

Interaction

SpatialIndex O(1) 히트 테스트, 툴팁, 박스/라소 셀렉션

scatter.vert (GLSL ES 3.0) 
layout(location = 0) in float a_x;
layout(location = 1) in float a_y;
layout(location = 2) in float a_colorData;

uniform mat4 u_projection;
uniform float u_pointSize;

void main() {
  gl_Position = u_projection * vec4(a_x, a_y, 0.0, 1.0);
  gl_PointSize = u_pointSize;
  v_colorData = a_colorData;
}

좌표 변환 시스템

3단계 좌표 변환을 통해 데이터 좌표가 화면에 매핑됩니다.

  • Screen → World: 마우스 좌표를 데이터 공간으로 변환
  • World → NDC: 카메라 매트릭스로 정규화 장치 좌표 변환
  • Separate Buffers: x, y를 분리 버퍼로 업로드 (인터리브 제거)
  • Palette Texture: 256x1 RGBA32F 텍스처에서 색상 룩업

사용법

React 컴포넌트로 몇 줄이면 시작할 수 있습니다.

BasicScatter.tsx 
import { ScatterChart } from '@seegak/react';

function MyVisualization() {
  const data = {
    x: new Float32Array([1, 2, 3, 4, 5]),
    y: new Float32Array([2, 4, 1, 5, 3]),
    labels: ['A', 'A', 'B', 'B', 'A'],
    colors: ['#e41a1c', '#e41a1c',
             '#377eb8', '#377eb8', '#e41a1c'],
  };

  return (
    <ScatterChart
      data={data}
      pointSize={3}
      opacity={0.85}
      tooltip toolbar legend autoFit
    />
  );
}

선언적 API

React 컴포넌트에 데이터를 전달하기만 하면 됩니다. WebGL 초기화, 셰이더 컴파일, 버퍼 관리는 모두 자동입니다.

  • data: Float32Array x, y + 라벨/컬러
  • tooltip: 호버 시 데이터 정보 표시
  • toolbar: Pan, Box Select, Lasso 도구
  • legend: 클러스터별 범례 (클릭으로 토글)
  • autoFit: 데이터 범위에 맞게 자동 줌

핵심 최적화 기법

Seegak이 빠른 이유

GPU

팔레트 텍스처

100K 포인트의 RGBA 색상 버퍼(1.6MB) 대신, 256x1 팔레트 텍스처(1KB) + Uint16 인덱스 버퍼(200KB)로 GPU 업로드를 10배 감소. 색상 변경 시에도 팔레트만 업데이트.

CPU

분리 버퍼 직접 업로드

data.x, data.y를 별도의 GL 어트리뷰트로 직접 업로드. 100K 포인트의 CPU 인터리브 루프를 완전히 제거하여 O(N) → O(0) CPU 작업으로 단축.

React

명령형 우회

React의 비동기 상태 스케줄링을 우회하여 instance.update()로 즉시 GPU에 데이터를 전달. setState → useEffect → render 파이프라인의 오버헤드를 제거.

벤치마크 분석

12가지 프로토콜 실험에서 얻은 핵심 인사이트

🏆 Zero-copy가 승리

0.0 ms parse

Arrow IPC, FlatBuffers, REST Binary 모두 파싱 시간이 0ms. TypedArray 뷰를 그대로 사용하는 zero-copy 방식이 직렬화/역직렬화 오버헤드를 완전히 제거합니다.

📦 JSON은 확실한 병목

+63% 느림

REST JSON(2541ms)은 2위 대비 981ms 느림. 10.85MB 문자열 직렬화 + 35ms 파싱이 원인. 대규모 수치 데이터에서 JSON은 최악의 선택입니다.

📊 압축의 트레이드오프

Zarr(1.70MB)가 가장 작지만 디컴프레스 비용(14.9ms)이 추가됨. Brotli(2.51MB)는 압축률이 좋지만 브라우저 해제 대기 시간이 존재. 로컬 환경에서는 비압축 바이너리가 유리합니다.

Zarr 1.70MB FlatBuf 2.09MB Brotli 2.51MB JSON 10.85MB

localhost 벤치마크가 놓치는 것

❓ 왜 모든 프로토콜이 비슷할까?

1560~1670 ms

로컬 환경(localhost)에서는 네트워크 레이턴시가 0ms, 대역폭이 무제한입니다. 데이터 크기 차이가 전송 시간에 반영되지 않아 10.85MB JSON과 1.70MB Zarr의 전송 시간이 거의 동일하게 측정되었습니다.

🌍 실제 배포 환경은 다르다

K-MAP 같은 실제 서비스에서는 서버가 원격에 있습니다. 네트워크 레이턴시(5~100ms)와 대역폭 제한(1.5~30Mbps)이 존재하며, 데이터 크기가 곧 전송 시간이 됩니다.

🛠 Docker Proxy 시뮬레이션

이 문제를 해결하기 위해 네트워크 시뮬레이션 프록시를 구축했습니다. Docker Compose에 역방향 프록시 컨테이너를 추가하여 RTT 레이턴시와 대역폭 제한을 적용합니다.

Fast WiFi: 5ms, 30Mbps 4G: 30ms, 10Mbps 3G: 100ms, 1.5Mbps

📐 아키텍처

Browser → Proxy (aiohttp :5002) → Backend (FastAPI :5001)
프록시가 요청/응답에 지연을 추가하고, 응답 크기에 비례하여 대역폭 제한을 시뮬레이션합니다. HTTP, WebSocket, SSE 모두 지원합니다.

네트워크 환경에서의 결론

Fast WiFi 시뮬레이션 — 로컬 vs 실제 네트워크에서 프로토콜 선택이 달라진다

🏆 압축이 핵심이다

Zarr 310.9 ms

로컬에서 중간이던 Zarr Chunked(766KB)가 네트워크 환경에서 1위로 등극. 가장 작은 데이터 크기가 대역폭 제한 환경에서 결정적 우위를 가집니다. 로컬 대비 5.2배 빠름.

❌ Arrow IPC의 반전

1위 → 7위

로컬에서 1위(1560ms)였던 Arrow IPC가 네트워크에서 773ms로 7위로 하락. 2.44MB의 비압축 데이터가 대역폭 제한에서 불리하게 작용합니다.

💡 데이터 크기 = 전송 시간

네트워크 환경에서는 데이터 크기와 전송 시간이 거의 비례합니다. 압축 프로토콜(Zarr, Brotli, Parquet)이 상위권을 차지하고, 대용량 비압축(JSON, Arrow)이 하위권으로 밀립니다.

Zarr 766KB → 1위 FlatBuf 1.0MB → 2위 Brotli 1.2MB → 3위

📋 결론: 환경에 따라 최적 프로토콜이 다르다

로컬/내부망: zero-copy (Arrow IPC, FlatBuffers) — 파싱 비용 최소화
원격/공용망: 압축 (Zarr, Brotli, Parquet) — 전송 크기 최소화
범용: FlatBuffers — 작은 크기 + zero-copy로 양쪽에서 모두 상위권
